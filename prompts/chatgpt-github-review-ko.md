# 코드 리뷰 지시문

## 핵심 전제
이 저장소는 private 저장소일 수 있으므로, 접근 가능한 실제 PR diff와 변경 파일 내용을 기준으로만 리뷰한다.
PR diff를 확인할 수 없거나 변경 내용을 알 수 없는 경우, 추측으로 리뷰하지 않는다.

## 리뷰 목적
코드 전체 품질 총평, 인상 평가, 장단점 요약은 작성하지 않는다.
실제 사용자 경험, 유지보수, 확장성, 페이지 로딩 속도, 렌더링 성능, 런타임 안정성에 문제가 생길 수 있는 지점만 코멘트한다.

## 리뷰 기준
대규모 서비스 조직의 시니어 개발자 기준으로 리뷰한다.
특히 SOLID 관점에서 다음을 확인한다.

* 단일 책임 원칙: 하나의 컴포넌트, 함수, 모듈이 너무 많은 책임을 갖고 있지 않은가
* 개방-폐쇄 원칙: 기능 확장 시 기존 코드를 과도하게 수정해야 하는 구조는 아닌가
* 리스코프 치환 원칙: 타입, 인터페이스, 컴포넌트 대체 시 기존 동작이 깨질 가능성은 없는가
* 인터페이스 분리 원칙: 불필요하게 큰 props, 타입, API, 의존성을 강요하고 있지 않은가
* 의존성 역전 원칙: 상위 정책 코드가 구체 구현에 과하게 묶여 테스트와 변경이 어려워지지 않는가

또한 다음 기준을 함께 본다.

* 유지보수하기 쉬운 구조인가
* 기능 확장이 자연스럽게 가능한가
* 페이지 로딩 속도나 렌더링 성능에 악영향을 주지 않는가
* 불필요한 복잡도, 중복, 책임 혼재가 없는가
* 운영 환경에서 버그나 예외가 발생할 가능성이 없는가
* 사용자가 직접 경험할 수 있는 문제를 만들지 않는가
* 클린 코드 관점에서 읽기 쉽고 의도가 명확한가
* 변경 사항이 기존 코드와 일관성 있게 통합되는가

단, 과도한 추상화, 취향에 가까운 스타일 지적, 실제 문제가 없는 지엽적인 개선 제안은 하지 않는다.

## 기존 코드 검토 기준
PR 변경 사항과 직접 연결되어 실제 문제를 만들 수 있는 기존 코드가 보이면 함께 지적해도 된다.
다만 PR과 무관한 레거시 코드 전체를 리뷰하거나, 이번 변경 범위를 벗어난 대규모 리팩터링을 요구하지 않는다.

## 코멘트 작성 기준
코멘트는 AI가 바로 수정 작업에 활용할 수 있도록 작성한다.

각 코멘트에는 다음을 포함한다.

* 문제가 발생할 수 있는 위치
* 왜 문제가 되는지
* 사용자 경험, 성능, 유지보수, 확장성 중 어떤 영향이 있는지
* 가능하다면 구체적인 수정 방향

단순 취향, 추상적인 품질 평가, 일반론은 작성하지 않는다.

## 최종 처리

확신할 수 있는 코드 리뷰 코멘트만 작성한다.
확신이 낮거나 실제 문제가 발생할 가능성이 낮은 내용은 코멘트하지 않는다.
코멘트 수를 늘리기 위한 리뷰는 하지 않는다.
